# TG-бот тренажёра обратной связи — подробная документация проекта

## 1. Что это за проект

Этот проект — Telegram-бот для тренировки навыка обратной связи в формате диалога с ИИ.

Базовая идея такая:
- пользователь заходит в бота;
- выбирает игру / метод обратной связи;
- выбирает конкретный сценарий внутри игры;
- общается с ИИ-агентом в рамках заданной роли;
- завершает диалог вручную кнопкой **«Дай обратную связь»** или автоматически по таймеру;
- система анализирует **весь** диалог и возвращает результаты аналитики.

Проект задуман как расширяемая система, где:
- можно добавлять новые игры;
- можно добавлять новые игровые сценарии;
- можно добавлять аналитические промты;
- тексты интерфейса можно менять через админку;
- результаты диалогов и аналитики сохраняются в базу.

---

## 2. Главная бизнес-идея

Проект состоит из двух больших частей:

### 2.1. Игровая часть для пользователя
Пользователь тренирует коммуникацию через диалог с ИИ:
- бот показывает условия сценария;
- при необходимости показывает изображение;
- ИИ отвечает в рамках конкретного промта;
- бот хранит контекст последних 15 пар сообщений для ИИ;
- весь диалог целиком записывается в базу.

### 2.2. Аналитическая часть
После завершения диалога:
- бот берёт **весь диалог по `dialog_id`**;
- собирает его в единый блок текста в правильном порядке;
- прогоняет этот текст через набор аналитических промтов для текущей игры;
- получает по каждому промту JSON-ответ с оценкой и текстом;
- показывает пользователю все результаты;
- считает общую сумму баллов;
- сохраняет результаты в таблицу пользовательских результатов.

---

## 3. Основные роли в системе

### 3.1. Пользователь
Обычный пользователь бота:
- запускает `/start`;
- при первом запуске вводит имя;
- выбирает игру;
- проходит диалог;
- получает аналитику.

### 3.2. Администратор
Администратор может:
- войти в админку по паролю;
- менять тексты интерфейса;
- менять пароль админки;
- добавлять игры;
- добавлять и удалять игровые промты;
- активировать и деактивировать промты;
- добавлять, редактировать и удалять аналитические промты.

### 3.3. Гостевой админ
Есть отдельный «гостевой» режим:
- вход по паролю `111111`;
- позволяет зайти в админку;
- позволяет смотреть меню и проходить почти все сценарии;
- **не сохраняет изменения в базу**.

Это режим безопасного ознакомления с админкой.

## 6. Общая логика пользовательского сценария

### 6.1. Первый запуск
При первом `/start`:
1. бот проверяет, есть ли `user_id` в таблице пользователей;
2. если нет — создаёт пользователя;
3. просит ввести имя:
   - **«Пожалуйста, введите своё имя (это имя будет использоваться только в боте)»**;
4. после ввода имени показывает обычный стартовый экран.

### 6.2. Обычный стартовый экран
На стартовом экране пользователь видит:
- приветственное сообщение;
- список игр первого уровня;
- кнопки-заглушки:
  - «Энциклопедия»
  - «Личный кабинет»

Кроме того, в текущем техническом режиме на экран выводится служебная информация:
- `chat_id`
- `user_id`
- данные пользователя Telegram

Эта часть помечена как техническая и должна быть отключена в релизе.

### 6.3. Выбор игры
Пользователь выбирает игру первого уровня, например:
- **Я-высказывание**

После этого открывается меню сценариев второго уровня, например:
- Подчинённый
- Коллега

### 6.4. Запуск игрового сценария
После выбора сценария:
1. бот показывает условия (`conditions`);
2. проверяет изображение:
   - если `img_id` есть — отправляет изображение по Telegram file_id;
   - если `img_id` нет, но есть `img_path` — загружает изображение, получает `img_id` и сохраняет его;
   - если ничего нет — шаг пропускается;
3. показывает приветствие сценария;
4. переходит в режим игрового диалога.

### 6.5. Ведение диалога
Пока диалог активен:
- пользователь отправляет сообщение;
- сообщение сохраняется в `dialog_messages`;
- бот показывает `[Думаю...]`;
- ИИ получает:
  - системный игровой промт;
  - последние 15 пар сообщений (30 реплик максимум);
- бот получает ответ ИИ;
- удаляет `[Думаю...]`;
- при необходимости снова показывает изображение сценария;
- отправляет ответ ИИ;
- показывает кнопку **«Дай обратную связь»**.

### 6.6. Завершение диалога
Диалог завершается двумя способами:
1. пользователь нажал кнопку **«Дай обратную связь»**;
2. истёк таймер (по умолчанию 10 минут / 600 секунд).

Оба пути должны приводить к одному и тому же результату:
- запуск аналитики;
- вывод результатов;
- возврат пользователя на стартовый экран.

## 7. Как устроена аналитика

### 7.1. Что именно анализируется
Для анализа берётся **не урезанная память ИИ**, а **весь диалог из базы** по `dialog_id`.

То есть:
- выбираются все записи из `dialog_messages` для данного `dialog_id`;
- сортируются по времени (`created_at`) и, как запасной вариант, по `id`;
- из них собирается единый текст.

### 7.2. Формат общего текста диалога
Общий текст строится так:
- `Пользователь сказал - ...`
- `ИИ ответил - ...`
- и так далее для всего диалога.

Последовательность обязана строго соблюдаться.

### 7.3. Отладка
Полученный полный блок диалога выводится в консоль.

Этот участок должен быть специально помечен:
- **ОТЛАДОЧНАЯ ИНФОРМАЦИЯ — В РЕЛИЗЕ ЗАКОММЕНТИТЬ**

### 7.4. Как запускается аналитика
Для текущей игры:
1. из таблицы аналитики выбираются все аналитические промты по `game_x`;
2. каждый промт по очереди получает полный текст диалога;
3. ИИ возвращает JSON:
   - `rating`
   - `text`
4. результат сохраняется во временный список;
5. параллельно каждый результат пишется в таблицу пользовательских результатов.

### 7.5. Что видит пользователь
Пока идёт анализ, пользователь видит сообщение:
- **«Идёт анализ. Это может занять несколько минут, пожалуйста, подождите»**

После завершения анализа:
- это сообщение удаляется;
- результаты выводятся по одному;
- между сообщениями делается задержка `0.2` секунды;
- в конце показывается общая сумма баллов;
- затем пользователь возвращается в `/start`.

## 8. Админка: для чего она нужна

Админка — это внутренний центр управления ботом.

Через неё можно:
- менять тексты интерфейса;
- управлять играми;
- управлять игровыми промтами;
- управлять аналитическими промтами;
- менять пароль;
- работать в гостевом режиме без сохранения изменений.

---

## 9. Вход в админку

### 9.1. Основной вход
Команда:
- `/admin`

Далее бот запрашивает пароль.

### 9.2. Ограничение по пользователю
Полноценный доступ в админку предназначен для пользователя с Telegram id:
- `467116941`

Если пароль верный, но `user_id` не тот:
- вход запрещается;
- пользователь возвращается на `/start`;
- в таблицу инцидентов записывается попытка несанкционированного входа.

### 9.3. Гостевой режим
Если пользователь вводит пароль:
- `111111`

то он попадает в гостевой режим.

Особенности гостевого режима:
- можно смотреть разделы админки;
- можно нажимать кнопки и проходить сценарии админки;
- изменения **не сохраняются**.

Приветствие гостевого режима:
- **«Это гостевой сеанс. Доступен просмотр, не доступны изменения»**

## 11. Игры и сценарии

### 11.1. Игры первого уровня
Игры первого уровня живут в таблице `games`.

Пример:
- `name = "Я-высказывание"`
- `game_id = "game_0"`

### 11.2. Сценарии второго уровня
Сценарии второго уровня описываются сочетанием:
- кнопок в `ui_texts`;
- промтов в `game_prompts`.

Кнопка второго уровня знает:
- к какой игре относится (`game`);
- в каком порядке показывается (`order`);
- какой игровой промт запускает (`game_alias`).

### 11.3. Пример логики связи
Например:
- игра: `game_0`
- кнопка: `Подчинённый`
- `game_alias = game_alias_subordinate`
- в `game_prompts.alias` лежит этот же `game_alias_subordinate`

Именно так кнопка связывается со сценарием.

## 12. Аналитические промты

Аналитические промты хранятся отдельно от игровых.

Каждая запись описывает:
- для какой игры она работает;
- как называется;
- как выглядит короткий header;
- какой у неё alias;
- какой у неё короткий comment;
- какой текст аналитического промта нужно отправить ИИ.

### 12.1. Новая аналитика
Сценарий:
1. админ выбирает игру;
2. вводит аналитический промт;
3. бот отправляет текст в ИИ;
4. ИИ возвращает:
   - `comment`
   - `alias`
   - `header`
5. бот создаёт запись в таблице аналитики.

### 12.2. Изменить аналитику
Сценарий:
1. показывается список аналитик в формате `comment | header`;
2. админ выбирает запись;
3. вводит новый текст промта;
4. ИИ возвращает только:
   - `header`
   - `comment`
5. `alias` остаётся прежним;
6. промт, header и comment перезаписываются.

### 12.3. Удалить аналитику
Сценарий:
1. показывается список аналитик;
2. админ выбирает одну запись;
3. бот спрашивает подтверждение;
4. при подтверждении аналитический промт удаляется;
5. перед удалением текст промта уходит в архивную таблицу удалённых промтов.

## 14. Работа с изображениями

Игровой сценарий может содержать картинку.

Подход такой:
- если в `game_prompts.img_id` уже есть Telegram file_id, бот использует его;
- если `img_id` нет, но есть `img_path`, бот загружает картинку, получает `img_id` и сохраняет его;
- если нет ни того, ни другого — сценарий продолжается без изображения.

Это удобно, потому что повторные отправки работают быстрее через file_id.

## 15. Хранение контекста для ИИ

У системы есть два уровня памяти:

### 15.1. Короткая память
Для ответа ИИ используется только последние 15 пар сообщений:
- то есть 30 последних реплик;
- это отправляется в ИИ как история диалога.

### 15.2. Полная память
Для аналитики используется **весь диалог** из таблицы `dialog_messages`.

Это важно:
- ИИ в моменте отвечает по ограниченному контексту;
- аналитика оценивает полную сессию целиком.

Для каждого сценария — своё состояние:
- вход по паролю;
- редактирование текстов;
- изменение пароля;
- добавление игры;
- работа с промтами;
- новая аналитика;
- изменение аналитики;
- удаление аналитики.

FSM нужен для того, чтобы бот понимал:
- что именно сейчас делает пользователь;
- какое сообщение от него ожидается;
- куда возвращать его дальше.

## 19. Как таблицы связаны между собой

Сейчас связи в проекте в основном **логические**, а не через жёсткие `foreign_key`.

Ключевые связи:
- `games.game_id` → используется в `game_prompts.game_id`
- `games.game_id` → используется в `ui_texts.game`
- `games.game_id` → используется в `dialog_messages.game_id`
- `games.game_id` → используется в `analytics_prompts.game`
- `game_prompts.alias` → используется в `ui_texts.game_alias`
- `game_prompts.alias` → используется в `dialog_messages.subgame_id`
- `dialog_messages.dialog_id` → объединяет один полный диалог
- `user_results.dialog_id` → связывает результат аналитики с конкретным диалогом

Иными словами:
- `games` — это верхний уровень;
- `game_prompts` — сценарии внутри игры;
- `dialog_messages` — фактическая история прохождения;
- `analytics_prompts` — правила анализа;
- `user_results` — результаты анализа для пользователя.

---

## 20. Почему сейчас не всё через foreign key

На текущем этапе проект развивается быстро, и часть связей оставлена логическими.

Плюсы такого подхода на раннем этапе:
- проще часто менять схему;
- проще не ломать миграции;
- быстрее двигаться по функционалу.

Минусы:
- выше риск появления «висячих» записей;
- целостность данных контролируется кодом, а не БД.

Для production-версии рекомендуется позже укрепить связи хотя бы для ключевых таблиц.

## 21. Что уже хорошо продумано в архитектуре

### 21.1. Отделение интерфейса от логики
UI-тексты вынесены в отдельную таблицу, поэтому:
- тексты можно менять без правки кода;
- кнопки можно переименовывать через админку.

### 21.2. Отделение игр от сценариев
Есть отдельная сущность игры и отдельная сущность сценария.

Это позволит:
- добавлять новые методы обратной связи;
- внутри каждой игры держать много сценариев.

### 21.3. Отделение игровых промтов от аналитических
Это правильное разделение:
- игровые промты отвечают за поведение ИИ в диалоге;
- аналитические промты отвечают за оценку завершённого диалога.

### 21.4. Хранение полной истории
Даже если ИИ видит только ограниченную память, система хранит полную историю и может делать полноценную аналитику.

## 22. Что важно помнить при развитии проекта

### 22.1. Репозиторий и локальный код могут расходиться
В течение разработки часть изменений могла уже обсуждаться и быть реализована локально, но ещё не оказаться в GitHub.

### 22.2. Документация должна обновляться вместе с логикой

- новый FSM-сценарий;
- новая роль;
- новый пользовательский раздел.

### 22.3. Текущая версия — это уже не «просто бот», а заготовка платформы
Проект уже тянет на каркас платформы тренажёров:
- пользователи;
- игры;
- сценарии;
- аналитика;
- результаты;
- админка;
- логирование.

## 23. Что ещё явно является следующим логичным развитием

На основе уже принятых решений проект естественно движется дальше в такие стороны:
- личный кабинет пользователя;
- просмотр истории прохождений;
- вывод результатов по играм;
- рейтинг / прогресс пользователя;
- экспорт результатов;
- улучшение аналитики;
- переход на более строгие связи БД;
- миграции вместо простого bootstrap-подхода.

Это не «обязательные» элементы прямо сейчас, а естественные следующие шаги из уже существующей архитектуры.

## 24. Краткое резюме проекта в одном абзаце

Это Telegram-бот-тренажёр, в котором пользователь проходит диалоги с ИИ по сценариям обратной связи, после чего система анализирует весь диалог через набор аналитических промтов, показывает результаты и сохраняет их. Управление играми, промтами, текстами интерфейса и аналитикой вынесено в админку. Архитектура уже построена так, чтобы проект можно было расширять: добавлять новые игры, новые сценарии, новые аналитики и в будущем развивать личный кабинет и прогресс пользователя.
